## 长期目标

建立以自主点餐门户为核心的完整营业链，通过以下方式**逐步摆脱对外卖平台的依赖**：

- **自主点餐** — 建立官网下单渠道，将平台抽成（15–30%）压缩到 Stripe 手续费（~3%）
- **品牌建设** — 通过官网、社交媒体、口碑形成独立品牌认知，不依赖平台流量
- **市场推广** — 线上广告、社区活动、促销优惠主动拉新，建立自有流量入口
- **客户沉淀** — 掌握自己的客户数据，通过会员积分、复购营销提升粘性；建立私域流量池（微信群、短信、邮件列表），直接触达客户，不受平台算法左右
- **自建配送** — 长期建立自有配送体系，彻底脱离平台生态；通过私域渠道直接派单，绕开平台调度
- **客户 App** — 提供比外卖平台更好的用户体验，增强品牌锁定

## 功能设计与开发 Timeline

> Claude 主导开发，Eli 负责审核和决策，每天投入约 2–3 小时。
> ⚠️ 下方为**原始计划**，部分已偏离实际，以上方「当前进度」为准。

### 准备期（3–5 天）
- 域名购买、GitHub / Supabase / Stripe 账号配置、项目结构搭建

### 阶段一 — 门户 + 自主点餐（2–3 周）
- 品牌介绍、故事、理念
- 菜单展示（图片、价格、描述）
- 在线点餐 + 加入购物车
- 在线支付（信用卡、Apple Pay、Google Pay）
- 自取流程：
  - 下单成功后自动生成 4 位取餐码，显示在确认页面并发短信给客人
  - 厨房平板/手机打开专用订单页面，新订单实时弹出并发出声音提醒
  - 厨房点击"确认接单" → 制作中；点击"餐已准备好" → 自动短信通知客人来取
  - 客人到店报取餐码，厨房点击"已取餐"完成订单
- 短信通知通过 Twilio 发送（约 $0.01/条）
- **订单自动记账** — 每笔订单支付成功后，自动写入 Supabase：订单金额、Stripe 手续费、净收入、时间戳、菜品明细，无需人工录入

### 阶段二 — 成熟运营（3–4 周）
- 会员/积分系统
- 客户数据库（复购、定向营销）
- 私域运营工具（短信/邮件营销、优惠券推送）
- 管理后台：
  - 菜单上下架、价格调整
  - 订单历史查询
  - 财务报表视图（调用阶段一积累的订单数据，按天/周/月汇总收入、Stripe 手续费、净收入）
  - 一键导出 CSV，供 QuickBooks / Wave 导入使用
- 数据分析（热销菜品、高峰时段、客单价趋势）

> 阶段一和二之间建议运营一段时间，收集真实反馈再启动。

### 阶段三 — 自建配送 + 客户 App（2–4 个月）

**客户 App**
- iOS / Android 原生 App 或 PWA（网站包装成 App，可装到手机桌面）
- 推送通知（订单状态、促销活动）
- 会员体系深度集成（积分、专属优惠）
- 实时配送追踪

**自建配送**
- 司机端 App/Web — 接单、导航、确认送达
- 调度系统 — 按距离/时间自动分配司机
- 配送费计算 — 按距离动态定价
- 私域派单 — 通过微信群/App 直接推送订单给司机，无需第三方平台
- 中期过渡方案：接入 DoorDash Drive（纯配送服务，按单付费不抽佣）

## 技术选型

| 层级 | 选择 | 运行在哪里 | 解释 |
|------|------|-----------|------|
| **前端** | Vite + React | 用户浏览器本地 | 打包成静态文件，下载到用户设备后在浏览器里运行，不占服务器资源 |
| **托管/部署** | GitHub Pages | GitHub 的 CDN 云服务器（全球分布） | 托管打包好的前端静态文件，免费，全球节点加速访问 |
| **CI/CD** | GitHub Actions | GitHub 云端 | 每次 push 代码自动触发构建+部署，无需手动操作 |
| **数据库** | Supabase | Supabase 云服务器（AWS 底层） | 托管的 PostgreSQL，附带实时推送和文件存储，无需管理服务器 |
| **支付函数** | Supabase Edge Function | Supabase 云端边缘节点 | 一小段 serverless 代码，只在被调用时运行，不需要常驻服务器 |
| **支付** | Stripe | Stripe 云服务器 | 第三方支付处理，信用卡数据完全在 Stripe 那边，不碰敏感数据 |
| **短信通知** | Twilio | Twilio 云服务器 | 发送取餐码、取餐提醒短信给客人，约 $0.01/条 |
| **代码管理** | GitHub | GitHub 云端 | 版本控制，Claude 可直接操作 |

**关键点：无自有服务器**，全部用第三方云服务，零运维成本。

## 数据流向

```
用户浏览器（React）
  ├── 读菜单/写订单 → Supabase 数据库（云）
  ├── 实时订单通知 ← Supabase Realtime（云）→ 厨房平板弹出 + 声音提醒
  ├── 短信通知（取餐码、餐好了）→ Twilio → 客人手机
  └── 点击付款
        → Supabase Edge Function（云）
              → Stripe（云）
                    → 银行卡扣款成功
                          → 订单数据自动写入 Supabase（金额、手续费、净收入、菜品明细）
                          → 返回结果给前端
```

## 核心原则

- **前端重，后端轻** — 尽量把逻辑放前端，后端只做前端做不了的事
- **工具越少越好** — 只引入真正需要的工具，降低维护成本
- **无自有服务器** — 全部 serverless/托管服务，零运维
- **分阶段推进** — 先上完整门户+点餐，再扩展运营功能，最终自建配送+App
- **架构先行** — 现在想清楚，避免后期技术债
- **Claude 全程深度参与** — 项目结构、命名、注释、文档始终保持清晰规范，让 Claude 能自主读懂、修改、测试、部署，尽量减少需要人工介入的环节；代码可维护性和易理解度是硬性要求，不因赶进度而妥协
